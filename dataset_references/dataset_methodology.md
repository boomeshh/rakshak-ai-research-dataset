# Dataset Methodology

## Overview

This document describes the methodology used to generate the Rakshak AI Research Dataset — a structured, synthetic cyber threat dataset designed to support AI/ML model development, NLP-based threat classification, rule-based risk scoring, dashboard analytics, and CERT escalation workflow testing for the **Rakshak AI Defence Cyber Safety Portal**.

---

## 1. Synthetic Data Generation

All records in this dataset are entirely synthetic. No real individuals, organisations, incidents, or systems are represented. The data was generated by hand and through structured templates to produce realistic-looking but entirely fictional cyber threat scenarios targeting Indian defence personnel, veterans, and their families.

### Generation Approach

- **Complaint records** were authored to reflect plausible Indian defence-related cyber threat narratives, including army welfare fraud, KYC scams, impersonation of defence officials, and banking fraud targeting veterans.
- **Phishing URLs** were constructed using common adversarial domain patterns (typosquatting, subdomain abuse, hyphenated impersonation) applied to fictional versions of Indian defence service names. All domains are non-existent and were verified to be fake at the time of authoring.
- **Scam messages** were written to mimic the tone, urgency, and contextual references found in real-world defence-targeted phishing campaigns, referencing programmes such as SPARSH, ECHS, CSD canteen, and Sainik welfare boards — without reproducing any real message content.
- **APK alert metadata** was constructed to represent plausible mobile malware scenarios impersonating Indian defence applications, with realistic package names, malware behaviour labels, and risk scores.
- **AI detection results** were generated to simulate the output of a threat classification model operating at approximately 87% accuracy, with `is_correct` values distributed accordingly.
- **Dashboard metrics** were computed from the underlying dataset records to ensure internal consistency (e.g., category counts summing to total threats detected).

### Realism Constraints

To ensure the dataset is useful for AI/ML training and testing, the following realism constraints were applied:

- All threat categories, risk levels, escalation statuses, and detection labels use consistent enumerated values defined in the global data model.
- Risk scores follow a realistic distribution: approximately 40% High, 40% Medium, and 20% Low.
- Timestamps are distributed across the year range 2023–2024 in ISO 8601 format.
- All cross-file references (complaint IDs in escalation logs, URLs in scam messages) are internally consistent.

## 2. Seeded Threat Simulation

To ensure reproducibility and consistency across dataset versions, a fixed seed and a set of governing parameters were used during data generation. These parameters define the statistical distribution targets and identifier formats applied throughout the dataset.

### Reproducibility Guarantee

Any researcher or engineer who regenerates this dataset using the seed parameters below and the generation methodology described in this document should produce a dataset with equivalent statistical properties, identifier formats, and distribution targets.

### Simulation Design

The seeded threat simulation approach involves:

1. **Fixing a global random seed** (`DATASET_SEED = 42`) to ensure deterministic ordering and sampling decisions.
2. **Applying distribution targets** for risk scores, escalation rates, and detection accuracy to govern the proportion of records in each category.
3. **Using sequential ID generation** starting from defined base values to produce consistent, sortable identifiers.
4. **Constraining the year range** to 2023–2024 to reflect a realistic and recent threat landscape.

All seed parameters are documented in [Section 7: Seed Parameters](#7-seed-parameters).

## 3. Manual Testing Methodology

After synthetic generation, all dataset files were reviewed manually to validate correctness, internal consistency, and realism.

### Review Process

1. **Schema validation** — Each JSON and CSV file was checked to confirm all required fields are present, non-null, and of the correct type for every record.
2. **Business rule verification** — Key business rules were verified manually:
   - All complaints with `cert_escalation_status = "Escalated"` have `risk_score >= 70`.
   - All `detection_label` values are correctly derived from `risk_score` (Malicious if > 70, Suspicious otherwise).
   - All `risk_level` values are consistent with their `risk_score` ranges (High: 75–100, Medium: 40–74, Low: 0–39).
3. **Cross-file consistency checks** — Identifiers referenced across multiple files were verified:
   - All `complaint_id` values in `cert_escalation_logs.txt` exist in `complaint_samples.json`.
   - All `complaint_id` values in `complaint_export.csv` match those in `complaint_samples.json`.
   - All URLs referenced in scam message files exist in `phishing_urls/phishing_links.txt`.
   - All domain entries in `malicious_domains.csv` are extracted from URLs in `phishing_links.txt`.
4. **Distribution verification** — Record counts and category distributions were checked against the targets defined in the seed parameters.
5. **Format compliance** — All timestamps were verified to conform to ISO 8601 (`YYYY-MM-DDTHH:MM:SSZ`), all IDs to their respective regex patterns, and all enum values to the global enumeration definitions.

### Automated Test Coverage

In addition to manual review, a suite of automated property-based and structural tests (located in the `tests/` directory) validates the dataset against all correctness properties defined in the design document. These tests cover:

- Schema completeness for all record types
- Business rule enforcement (escalation threshold, detection label derivation)
- Cross-file referential integrity
- Format compliance for IDs, timestamps, and risk scores
- Distribution targets (detection accuracy ≥ 85%, all threat categories represented)

## 5. Dataset Limitations

Researchers and engineers using this dataset should be aware of the following limitations:

### Coverage Limitations

- **Threat type coverage**: The dataset covers 5 threat categories (Phishing, APK_Malware, Impersonation, Banking_Fraud, OTP_Scam). Emerging or niche threat types are not represented.
- **Language coverage**: All scam messages and complaint descriptions are in English. Real-world Indian defence cyber threats frequently occur in Hindi, regional languages, and mixed-language (Hinglish) formats. This dataset does not capture multilingual threat patterns.
- **Channel coverage**: Scam messages cover SMS, WhatsApp, and Telegram. Other channels (email, voice calls, social media) are not included.
- **Temporal coverage**: All records are dated within 2023–2024. Threat patterns from earlier periods or future evolutions are not represented.

### Statistical Limitations

- **Small dataset size**: With 15 complaint records, 15 phishing URLs, 10 APK alerts, and 15 AI detection results, this dataset is suitable for pipeline testing and model prototyping but is too small for production model training without augmentation.
- **Controlled distributions**: Risk score and accuracy distributions are artificially controlled to meet targets (40/40/20 split, 87% accuracy). Real-world distributions may differ significantly.
- **No adversarial variation**: The dataset does not include adversarial examples, edge cases, or deliberately ambiguous records designed to stress-test model robustness.

### Realism Limitations

- **Synthetic narratives**: Complaint descriptions and scam messages, while contextually plausible, were authored rather than collected from real incidents. They may not capture the full linguistic diversity, grammatical patterns, or social engineering sophistication of real-world threats.
- **No temporal drift**: The dataset does not model how threat patterns evolve over time. All records reflect a static snapshot of threat types and tactics.
- **No ground truth provenance**: Unlike datasets derived from real threat intelligence feeds, this dataset has no external ground truth. All labels are assigned by the generation methodology, not validated against real-world outcomes.

### Usage Limitations

- This dataset should not be used as the sole training source for production AI/ML models deployed in real security contexts.
- Risk scores and detection labels are illustrative and should not be interpreted as calibrated probability estimates.
- The dataset is not suitable for benchmarking against real-world threat detection systems.

## 6. Recommended Usage Guidelines

### For AI/ML Engineers

- **Model prototyping**: Use this dataset to build and test threat classification pipelines, risk scoring models, and NLP-based scam detection systems before integrating real data.
- **Pipeline validation**: Use the structured JSON and CSV files to validate data ingestion, parsing, and preprocessing pipelines.
- **Schema testing**: Use the dataset to verify that your model's input schema handling is correct before connecting to live data sources.
- **Augmentation**: Treat this dataset as a seed for larger synthetic datasets. Use the seed parameters and generation methodology to produce additional records with consistent statistical properties.

### For QA Engineers

- **Dashboard testing**: Use `testing_exports/dashboard_metrics.json` to seed the Rakshak AI portal analytics view without requiring live data.
- **Escalation workflow testing**: Use `testing_exports/cert_escalation_logs.txt` and `fake_complaints/complaint_samples.json` to drive CERT escalation workflow validation end-to-end.
- **Regression testing**: Use the dataset as a stable, versioned fixture for regression tests. The seed parameters ensure the dataset can be regenerated consistently.

### For Researchers

- **Baseline evaluation**: Use this dataset to establish baseline performance metrics for threat classification models in the Indian defence cyber threat domain.
- **Methodology reference**: Use the seed parameters and generation approach as a template for creating larger, domain-specific synthetic datasets.
- **Citation**: When publishing research that uses this dataset, cite the Rakshak AI Research Dataset and note that all data is synthetic.

### General Guidelines

- Always read the [Disclaimer](#4-disclaimer) before using this dataset.
- Do not mix this synthetic dataset with real threat intelligence data without clearly labelling the provenance of each record.
- Do not deploy models trained solely on this dataset in production security systems.
- When extending the dataset, follow the seed parameters and schema constraints defined in this document to maintain cross-file consistency.
- Run the automated test suite (`pytest tests/ -v`) after any modifications to verify that all correctness properties are still satisfied.

## 7. Seed Parameters

The following seed parameters govern the generation of this dataset. They are documented here to ensure reproducibility and to allow researchers to regenerate or extend the dataset with consistent statistical properties.

```
DATASET_SEED              = 42
YEAR_RANGE                = 2023–2024
COMPLAINT_ID_START        = RKS-2024-0001
APK_ID_START              = APK-2024-001
DETECTION_ID_START        = DET-2024-0001
RISK_SCORE_DISTRIBUTION   = { High: 40%, Medium: 40%, Low: 20% }
ACCURACY_TARGET           = 0.87  (87% of is_correct = True)
ESCALATION_RATE           = 0.33  (1 in 3 high-risk complaints escalated)
```

### Parameter Descriptions

| Parameter | Value | Description |
|-----------|-------|-------------|
| `DATASET_SEED` | `42` | Global random seed for deterministic generation ordering and sampling decisions |
| `YEAR_RANGE` | `2023–2024` | All timestamps and date fields fall within this range |
| `COMPLAINT_ID_START` | `RKS-2024-0001` | First complaint ID in the sequential series; format: `RKS-YYYY-NNNN` |
| `APK_ID_START` | `APK-2024-001` | First APK alert ID in the sequential series; format: `APK-YYYY-NNN` |
| `DETECTION_ID_START` | `DET-2024-0001` | First AI detection result ID in the sequential series; format: `DET-YYYY-NNNN` |
| `RISK_SCORE_DISTRIBUTION` | High: 40%, Medium: 40%, Low: 20% | Target proportion of records at each risk level across complaint and APK datasets |
| `ACCURACY_TARGET` | `0.87` | Target proportion of AI detection records where `is_correct = True` (minimum threshold: 0.85) |
| `ESCALATION_RATE` | `0.33` | Target proportion of high-risk complaints with `cert_escalation_status = "Escalated"` |

### Risk Score Thresholds

| Risk Level | Score Range | Detection Label |
|------------|-------------|-----------------|
| High | 75–100 | Malicious (if score > 70) |
| Medium | 40–74 | Suspicious (if score ≤ 70) |
| Low | 0–39 | Suspicious |

> **Note:** The detection label threshold (> 70) and the risk level threshold (≥ 75 for High) are independent rules. A record with `risk_score = 72` is classified as Medium risk but labelled Malicious. Both rules apply independently across all dataset files.

### ID Format Specifications

| Entity | Format | Regex Pattern | Example |
|--------|--------|---------------|---------|
| Complaint | `RKS-YYYY-NNNN` | `RKS-\d{4}-\d{4}` | `RKS-2024-0001` |
| APK Alert | `APK-YYYY-NNN` | `APK-\d{4}-\d{3}` | `APK-2024-001` |
| Detection Result | `DET-YYYY-NNNN` | `DET-\d{4}-\d{4}` | `DET-2024-0001` |

### Timestamp Format

All timestamp fields use ISO 8601 UTC format: `YYYY-MM-DDTHH:MM:SSZ`

Date-only fields (e.g., `first_seen_date` in `malicious_domains.csv`) use: `YYYY-MM-DD`
